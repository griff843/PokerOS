# Canonical Drill Schema — Poker Coach OS

_Version: 1.0 | Sprint: SPRINT-DRILL-SCHEMA-FOUNDATION | Date: 2026-03-08_

---

## 1. Design Goals

The canonical drill schema is the **single source of truth** for all drill content in Poker Coach OS. Every drill — whether authored by hand, generated by AI, or imported from a live session — must conform to this schema.

The schema must:

1. **Unify CLI and Table Sim** — one drill format serves both interfaces, eliminating manual sync between two content files
2. **Support pool-aware exploit training** — drills can have different correct answers per opponent pool type
3. **Enable multi-street hand trees** — a single drill can represent linked decisions across flop, turn, and river
4. **Carry full scenario context** — board, hero hand, positions, stacks, action history are structured data, not buried in prompt text
5. **Preserve scoring compatibility** — the schema maps cleanly to both `scoreDrill()` (CLI) and `scoreTableSimDrill()` (Table Sim)
6. **Feed all downstream systems** — analytics, adaptive curriculum, AI coaching, leak detection all consume this schema
7. **Remain backward-compatible** — existing 30 drills can be migrated without data loss
8. **Scale to 1700+ drills** — no structural bottlenecks at target content volume

## 2. Schema Scope

### Supported drill types

| Type | Description | Status |
|---|---|---|
| Single-decision | One action at one street (current format) | Supported now |
| Pool-aware | Different correct answer per Pool A/B/C | New in canonical |
| Multi-street | Linked decisions across 2-3 streets | New in canonical |
| Table Sim visual | Full board/hand rendering with sizing | Supported now |
| Preflop | Pre-board decision (open, 3-bet, call, fold) | New in canonical |
| Live-session imported | Drill generated from a logged hand | Future — schema ready |

### What is NOT in scope

- Range visualization data (separate system, references drills by ID)
- Solver output attachments (future extension via metadata)
- Audio/video content (out of scope for drill schema)

## 3. Canonical Top-Level Structure

```
CanonicalDrill {
  // ── Identity ──
  drill_id:             string        REQUIRED
  node_id:              string        REQUIRED
  version:              string        REQUIRED  default "1.0.0"

  // ── Display ──
  title:                string        REQUIRED
  prompt:               string        REQUIRED

  // ── Scenario ──
  scenario:             Scenario      REQUIRED

  // ── Decision ──
  decision_point:       DecisionPoint REQUIRED  (single-decision drills)
  options:              Option[]      REQUIRED  min 2

  // ── Answer ──
  answer:               Answer        REQUIRED  (baseline/GTO answer)
  answer_by_pool:       AnswerByPool  OPTIONAL  (pool-variant answers)

  // ── Tags ──
  tags:                 string[]      REQUIRED  min 1  (classification tags, category:value)
  difficulty:           integer       REQUIRED  1-5

  // ── Multi-Street Extension ──
  steps:                Step[]        OPTIONAL  (multi-street drills only)

  // ── Coaching ──
  coaching_context:     CoachingCtx   OPTIONAL

  // ── Metadata ──
  metadata:             Metadata      OPTIONAL
}
```

### Field Reference

| Field | Type | Required | Description |
|---|---|---|---|
| `drill_id` | string | Yes | Unique identifier. Descriptive: `srp_river_bluffcatch_01` |
| `node_id` | string | Yes | FK to parent node. Must match `^[a-z0-9_]+$` |
| `version` | string | Yes | Semver. Bumped when drill content changes |
| `title` | string | Yes | Short human-readable title (displayed in UI) |
| `prompt` | string | Yes | Full scenario text shown to the player |
| `scenario` | Scenario | Yes | Structured representation of the poker situation |
| `decision_point` | DecisionPoint | Yes | What the player is deciding at this moment |
| `options` | Option[] | Yes | Available action choices (min 2, max 5) |
| `answer` | Answer | Yes | Baseline/GTO correct answer |
| `answer_by_pool` | AnswerByPool | No | Pool-specific correct answers (keys: "A", "B", "C") |
| `tags` | string[] | Yes | Classification tags in `category:value` format (min 1) |
| `difficulty` | integer | Yes | 1 (basic) to 5 (advanced) |
| `steps` | Step[] | No | Multi-street decision sequence. Omit for single-decision drills |
| `coaching_context` | CoachingCtx | No | Extra context for AI coaching responses |
| `metadata` | Metadata | No | Authoring info, source, flags |

## 4. Scenario Model

The scenario is a structured representation of the poker situation at the moment of decision.

```
Scenario {
  game:              string        REQUIRED  default "NLHE Cash"
  street:            string        REQUIRED  enum: "preflop" | "flop" | "turn" | "river"
  pot_type:          string        REQUIRED  enum: "SRP" | "3BP" | "4BP" | "limp" | "squeeze" | "multiway"
  players_to_flop:   integer       REQUIRED  default 2
  hero_position:     string        REQUIRED  e.g. "BB", "BTN", "CO", "SB", "HJ", "UTG"
  villain_position:  string        REQUIRED  e.g. "BTN", "BB" (primary villain for HU)
  effective_stack_bb: number       OPTIONAL  effective stack in big blinds
  pot_size_bb:       number        OPTIONAL  pot size in big blinds at decision point
  board:             Board         OPTIONAL  (null for preflop drills)
  hero_hand:         [string, string]  OPTIONAL  (null for range-level drills)
  action_history:    ActionStep[]  OPTIONAL  sequence of actions leading to decision
}

Board {
  flop:  [string, string, string]  REQUIRED  card format: "Js", "7c", "3d"
  turn:  string | null             REQUIRED  null if decision is on flop
  river: string | null             REQUIRED  null if decision is on flop or turn
}

ActionStep {
  street:   string    REQUIRED  "preflop" | "flop" | "turn" | "river"
  player:   string    REQUIRED  "hero" | "villain" | "villain2" etc.
  action:   string    REQUIRED  "open", "call", "raise", "check", "bet", "fold"
  size_bb:  number    OPTIONAL  bet/raise size in big blinds
  size_pct_pot: number OPTIONAL  bet/raise size as % of pot
}
```

### Scenario design notes

- `board` is null for preflop drills (hero decides before seeing a flop)
- `hero_hand` is null for range-level drills where the player decides for a range, not a specific holding
- `action_history` captures the full line leading to this decision (e.g., "villain opens 3bb, hero 3-bets to 10bb, villain calls")
- `effective_stack_bb` is the effective stack at the START of the hand (not remaining stack at decision point)
- Card format uses standard 2-char notation: rank + suit (`2-9`, `T`, `J`, `Q`, `K`, `A` + `s`, `h`, `d`, `c`)

### Scenario for preflop drills

```json
{
  "game": "NLHE Cash",
  "street": "preflop",
  "pot_type": "SRP",
  "players_to_flop": 2,
  "hero_position": "CO",
  "villain_position": "BB",
  "effective_stack_bb": 100,
  "board": null,
  "hero_hand": ["Ah", "Ts"],
  "action_history": []
}
```

### Scenario for postflop drills

```json
{
  "game": "NLHE Cash",
  "street": "river",
  "pot_type": "SRP",
  "players_to_flop": 2,
  "hero_position": "BB",
  "villain_position": "BTN",
  "effective_stack_bb": 100,
  "pot_size_bb": 12,
  "board": {
    "flop": ["Js", "7c", "3d"],
    "turn": "2h",
    "river": "Jd"
  },
  "hero_hand": ["Jh", "9h"],
  "action_history": [
    { "street": "preflop", "player": "villain", "action": "open", "size_bb": 3 },
    { "street": "preflop", "player": "hero", "action": "call", "size_bb": 3 },
    { "street": "flop", "player": "hero", "action": "check" },
    { "street": "flop", "player": "villain", "action": "bet", "size_pct_pot": 50 },
    { "street": "flop", "player": "hero", "action": "call" },
    { "street": "turn", "player": "hero", "action": "check" },
    { "street": "turn", "player": "villain", "action": "check" },
    { "street": "river", "player": "villain", "action": "bet", "size_pct_pot": 75 }
  ]
}
```

## 5. Answer Model

```
Answer {
  correct:        string    REQUIRED  action name: "CALL", "FOLD", "RAISE", "BET", "CHECK", "OPEN", "3BET"
  accepted:       string[]  REQUIRED  alternative acceptable actions (may be empty)
  correct_size:   SizeSpec  OPTIONAL  correct sizing when sizing is relevant
  required_tags:  string[]  REQUIRED  min 1, rule tags from VALID_TAGS enum
  explanation:    string    REQUIRED  why this answer is correct
}

SizeSpec {
  size_bb:         number   OPTIONAL  absolute size in big blinds
  size_pct_pot:    number   OPTIONAL  size as % of pot
  size_bucket:     number   OPTIONAL  bucket index for Table Sim sizing buttons
  tolerance_pct:   number   OPTIONAL  acceptable deviation (default 15%)
}

AnswerByPool {
  "A": Answer      OPTIONAL  correct answer vs Pool A (competent regular)
  "B": Answer      OPTIONAL  correct answer vs Pool B (passive recreational)
  "C": Answer      OPTIONAL  correct answer vs Pool C (aggressive gambler)
}
```

### Answer design notes

- `correct` uses **action names** (not option keys like "A", "B"). This is a deliberate change from the current CLI format where `correct: "A"` references an option key. The canonical format decouples the answer from the option presentation order.
- `accepted` lists alternative actions that receive full action credit. Example: a spot where both CALL and RAISE are reasonable.
- `correct_size` is optional. When present and the interface supports sizing (Table Sim), the scoring engine evaluates sizing accuracy.
- `required_tags` must contain at least 1 rule tag from the `VALID_TAGS` registry.
- When `answer_by_pool` is present, the system resolves the answer based on the session's active pool. If no pool is selected (baseline), the top-level `answer` is used.
- Not every pool key needs to be present in `answer_by_pool`. Only include pools where the correct action genuinely differs from baseline.

### Answer resolution logic

```
function resolveAnswer(drill, activePool):
  if activePool and drill.answer_by_pool and drill.answer_by_pool[activePool]:
    return drill.answer_by_pool[activePool]
  else:
    return drill.answer
```

## 6. Tagging Model

Every drill carries two layers of tags as defined in `TAGGING_SYSTEM.md`.

### Rule Tags (scoring layer)

**Location**: `answer.required_tags` (and each `answer_by_pool.{pool}.required_tags`)

**Format**: Flat `snake_case` strings from the `VALID_TAGS` enum.

**Purpose**: The player must identify these concepts during the drill. Missed tags reduce the score (30% weight).

**Validation**: Must be members of the `VALID_TAGS` array at content load time.

**Minimum**: 1 per answer variant.

### Classification Tags (analytics layer)

**Location**: `tags[]` at the top level of the drill.

**Format**: `category:value` (colon-separated).

**Purpose**: Categorize the drill for analytics, curriculum, filtering, and AI coaching. Not used in scoring. Players never see or interact with these.

**Categories**: `street`, `pot`, `position`, `spot`, `board`, `concept`, `decision`, `pool`

**Validation**: Category prefix must be registered. Values are freeform within each category.

**Minimum**: 1 per drill (practically, 3-6 per drill for useful analytics).

### Authoring rules

1. Every drill must have at least 1 rule tag in `answer.required_tags`
2. Every drill should have 3-6 classification tags in `tags[]`
3. If `answer_by_pool` entries have different `required_tags`, the pool-specific tags train exploit recognition (e.g., `underfold_exploit` only appears in Pool B's answer)
4. Classification tags should cover: street, pot type, position, at least one concept, and the decision type
5. Pool-specific drills should include the relevant `pool:pool_x` classification tag

### Example tag usage

```json
{
  "tags": [
    "street:river",
    "pot:srp",
    "position:oop",
    "spot:btn_vs_bb",
    "board:paired",
    "concept:blocker_effect",
    "decision:bluff_catch"
  ],
  "answer": {
    "required_tags": ["paired_top_river"]
  },
  "answer_by_pool": {
    "B": {
      "required_tags": ["paired_top_river", "underfold_exploit"]
    }
  }
}
```

## 7. Multi-Street Extension Model

### Architecture: Embedded Steps

Multi-street drills use an embedded `steps[]` array within the same drill object. Each step represents one decision point at one street.

```
Step {
  step_id:         string        REQUIRED  e.g. "flop", "turn", "river" or "step_1", "step_2"
  street:          string        REQUIRED  "flop" | "turn" | "river"
  prompt:          string        REQUIRED  situation text for this specific decision
  board_update:    BoardUpdate   OPTIONAL  new card(s) revealed for this step
  decision_point:  DecisionPoint REQUIRED  what the player decides
  options:         Option[]      REQUIRED  available choices
  answer:          Answer        REQUIRED  correct answer for this step
  answer_by_pool:  AnswerByPool  OPTIONAL  pool variants for this step
}

BoardUpdate {
  turn:  string | null   card dealt on turn (for flop -> turn transition)
  river: string | null   card dealt on river (for turn -> river transition)
}
```

### Why embedded steps (not linked drills)

| Approach | Pros | Cons |
|---|---|---|
| **Embedded steps** | Self-contained, one SRS entry, full context | Larger drill objects |
| Linked drills | Simpler objects, reusable steps | Complex relationships, harder to author, SRS ambiguity |
| Sequence model | Clean separation | Extra indirection, three entities to manage |

**Decision: Embedded steps.** Reasons:

1. A multi-street hand is conceptually ONE training item — the SRS should schedule the whole hand
2. The AI coach needs full hand context across all streets to give good explanations
3. Content authoring is self-contained — no cross-referencing between files
4. Scoring can be composite (average across steps) or per-step, with no join queries
5. The existing content loader can handle it — just a larger drill object

### Single-decision vs multi-street

- **Single-decision drill**: Top-level `decision_point`, `options`, `answer`, `answer_by_pool` are used. `steps` is omitted or null.
- **Multi-street drill**: `steps[]` is present and contains 2+ Step objects. The top-level `decision_point` and `answer` are **not used** (set to the first step's values for backward compatibility, or omitted). The system iterates through steps sequentially.

### Scoring multi-street drills

```
Multi-street score = average of per-step scores

For each step:
  step_score = scoreDrill({ userAnswer, userTags, answer: step.answer })

Total score = sum(step_scores) / step_count
```

The pass threshold (0.6) applies to the total score. Individual step results are recorded for analytics (which street did the player struggle with?).

### Multi-street example structure

```json
{
  "drill_id": "srp_flop_to_river_bluffcatch_01",
  "steps": [
    {
      "step_id": "flop",
      "street": "flop",
      "prompt": "Flop: Js 7c 3d. Villain c-bets 50% pot. What do you do?",
      "decision_point": { "street": "flop", "facing": { "action": "bet", "size_pct_pot": 50 } },
      "options": [
        { "key": "CALL", "label": "Call" },
        { "key": "FOLD", "label": "Fold" },
        { "key": "RAISE", "label": "Raise" }
      ],
      "answer": {
        "correct": "CALL",
        "accepted": [],
        "required_tags": ["cbet_dry_flop"],
        "explanation": "Top pair on a dry flop. Call to keep villain's bluffs in."
      }
    },
    {
      "step_id": "turn",
      "street": "turn",
      "prompt": "Turn: 2h. Villain checks. What do you do?",
      "board_update": { "turn": "2h" },
      "decision_point": { "street": "turn", "facing": null },
      "options": [
        { "key": "CHECK", "label": "Check" },
        { "key": "BET", "label": "Bet" }
      ],
      "answer": {
        "correct": "CHECK",
        "accepted": ["BET"],
        "required_tags": ["equity_denial"],
```

## 8. Decision Point and Options

```
DecisionPoint {
  street:                string     REQUIRED  "preflop" | "flop" | "turn" | "river"
  facing:                Facing     OPTIONAL  null if hero acts first (opening, probing)
  sizing_buttons_enabled: boolean   OPTIONAL  default false (Table Sim feature)
}

Facing {
  action:       string    REQUIRED  "bet", "raise", "overbet", "3bet", "4bet", "open", "check"
  size_pct_pot: number    OPTIONAL  size as % of pot
  size_bb:      number    OPTIONAL  size in big blinds
}

Option {
  key:    string    REQUIRED  action name: "CALL", "FOLD", "RAISE", "BET", "CHECK", "OPEN", "3BET"
  label:  string    REQUIRED  human-readable display text
}
```

### Option design notes

- Option `key` uses **canonical action names** in uppercase. This is a deliberate change from the current CLI format where keys are arbitrary letters ("A", "B", "C", "D").
- The `key` field is what the scoring engine matches against `answer.correct` and `answer.accepted`.
- The `label` field is what the player sees. It can contain more context than the key (e.g., `label: "Raise to 3x"` with `key: "RAISE"`).
- This unification means CLI and Table Sim use the same answer evaluation logic.

## 9. Coaching Context

Optional structured data to help the AI coaching layer generate better responses.

```
CoachingContext {
  key_concept:       string    OPTIONAL  primary strategic concept being tested
  common_mistake:    string    OPTIONAL  what players typically get wrong at this spot
  range_context:     string    OPTIONAL  relevant range dynamics for this decision
  population_note:   string    OPTIONAL  how population tendencies affect this spot
  follow_up:         string    OPTIONAL  what to study next if this drill is missed
}
```

### Coaching context design notes

- This is author-provided context that supplements the explanation
- The AI coach uses this alongside the drill's structured data (answer, tags, scenario) to generate conversational feedback
- When absent, the AI coach works from the explanation text and structured data alone
- This field is optional — it enriches coaching quality but is not required for a drill to function

## 11. Table Sim Compatibility

The canonical drill schema is a superset of the current Table Sim schema. Here is how each Table Sim field maps:

### Field mapping

| Table Sim field | Canonical field | Notes |
|---|---|---|
| `drill_id` | `drill_id` | Same |
| `node_id` | `node_id` | Same |
| `title` | `title` | Same |
| `prompt` | `prompt` | Same |
| `meta.game` | `scenario.game` | Moved into scenario |
| `meta.players_to_flop` | `scenario.players_to_flop` | Moved into scenario |
| `meta.hero_pos` | `scenario.hero_position` | Renamed for consistency |
| `meta.villain_pos` | `scenario.villain_position` | Renamed for consistency |
| `board` | `scenario.board` | Moved into scenario |
| `hero_hand` | `scenario.hero_hand` | Moved into scenario |
| `pot_bb` | `scenario.pot_size_bb` | Renamed for clarity |
| `decision_point.street` | `decision_point.street` | Same |
| `decision_point.facing` | `decision_point.facing` | Same structure |
| `decision_point.options` | `options[].key` | Options promoted to top level |
| `decision_point.sizing_buttons_enabled` | `decision_point.sizing_buttons_enabled` | Same |
| `answer_key.correct_action` | `answer.correct` | Unified answer model |
| `answer_key.accepted_actions` | `answer.accepted` | Unified answer model |
| `answer_key.correct_size_bucket` | `answer.correct_size.size_bucket` | Nested in SizeSpec |
| `answer_key.required_tags` | `answer.required_tags` | Same |
| `answer_key.explanation` | `answer.explanation` | Same |

### Required fields for Table Sim rendering

The Table Sim requires these fields to render the poker table visually:

| Field | Why |
|---|---|
| `scenario.board` | Render community cards (null for preflop) |
| `scenario.hero_hand` | Render hero's hole cards |
| `scenario.hero_position` | Position label on table |
| `scenario.villain_position` | Position label on table |
| `decision_point.facing` | Display facing action and size |
| `decision_point.sizing_buttons_enabled` | Show/hide sizing selector |
| `options` | Render action buttons |

Preflop drills where `scenario.board` is null will render the table with face-down community cards and hero's hole cards visible.

### Board texture detection

Board texture detection (`apps/table-sim/src/lib/board-texture.ts`) operates on `scenario.board.flop`, `scenario.board.turn`, `scenario.board.river` — the canonical schema provides these in the same card format already used.

## 12. Backward Compatibility

### Existing content

| Content | Location | Count | Format |
|---|---|---|---|
| CLI drills | `content/drills/hu_seed.json` | 30 | `DrillFileEntrySchema` |
| Table Sim drills | `apps/table-sim/public/content/drills.json` | 30 | `TableSimDrillSchema` |

These are the same 30 drills in two different formats.

### Migration path

Existing drills can be migrated to the canonical format with these transformations:

**From CLI format:**

| CLI field | Canonical field | Transform |
|---|---|---|
| `drill_id` | `drill_id` | Keep |
| `node_id` | `node_id` | Keep |
| (missing) | `version` | Set to "1.0.0" |
| (missing) | `title` | Copy from Table Sim version or derive from node name |
| `prompt` | `prompt` | Keep |
| (missing) | `scenario` | Extract from Table Sim version |
| (missing) | `decision_point` | Extract from Table Sim version |
| `options[].key` | `options[].key` | **Transform**: "A" -> action name from label (e.g., "A"/"Call" -> "CALL") |
| `answer.correct` | `answer.correct` | **Transform**: key letter -> action name ("A" -> "CALL") |
| `answer.accepted` | `answer.accepted` | **Transform**: key letters -> action names |
| `answer.explanation` | `answer.explanation` | Keep |
| `answer.required_tags` | `answer.required_tags` | Keep (already rule tags) |
| `tags` | `tags` | **Transform**: flat tags -> `category:value` format |
| `difficulty` | `difficulty` | Keep |

**From Table Sim format:**

| Table Sim field | Canonical field | Transform |
|---|---|---|
| `meta.*` | `scenario.*` | Restructure into scenario |
| `board` | `scenario.board` | Move under scenario |
| `hero_hand` | `scenario.hero_hand` | Move under scenario |
| `pot_bb` | `scenario.pot_size_bb` | Rename |
| `decision_point.options` | `options` | Promote to top level, add labels |
| `answer_key.*` | `answer.*` | Restructure field names |

### Migration strategy

1. Write a one-time migration script that merges the CLI and Table Sim formats into canonical format
2. The canonical file becomes the single source at `content/drills/`
3. The content loader and Table Sim are updated to read the canonical format
4. Legacy `DrillFileEntrySchema` and `TableSimDrillSchema` remain in code temporarily with adapter functions
5. Once all consumers are migrated, legacy schemas are removed

### Node ID compatibility

Existing node IDs (`hu_01` through `hu_10`) remain valid under the relaxed `^[a-z0-9_]+$` convention. No node migration needed.

## 13. Validation Strategy

### Required fields (all drill types)

```
drill_id              non-empty string, unique
node_id               non-empty string, matches ^[a-z0-9_]+$
version               semver string
title                 non-empty string
prompt                non-empty string
scenario.game         non-empty string
scenario.street       enum: preflop | flop | turn | river
scenario.pot_type     enum: SRP | 3BP | 4BP | limp | squeeze | multiway
scenario.players_to_flop  integer >= 2
scenario.hero_position    non-empty string
scenario.villain_position non-empty string
decision_point.street     enum: preflop | flop | turn | river
options               array, min 2 items
options[].key         non-empty string (action name)
options[].label       non-empty string
answer.correct        non-empty string (must match an option key)
answer.required_tags  array, min 1 item (each must be in VALID_TAGS)
answer.explanation    non-empty string
tags                  array, min 1 item (each must have registered category prefix)
difficulty            integer 1-5
```

### Conditional requirements

| Condition | Required fields |
|---|---|
| `scenario.street != "preflop"` | `scenario.board` (with `flop` of 3 cards) |
| `scenario.street == "turn"` | `scenario.board.turn` must be non-null |
| `scenario.street == "river"` | `scenario.board.turn` and `scenario.board.river` must be non-null |
| `decision_point.sizing_buttons_enabled == true` | `answer.correct_size` must be present |
| `steps` is present | Must have 2+ steps, each with required Step fields |
| `answer_by_pool` is present | Each pool entry must be a valid Answer |

### Invalid states

- `answer.correct` does not match any `options[].key`
- `answer.required_tags` contains a tag not in `VALID_TAGS`
- `tags[]` contains an entry without a `:` separator
- `tags[]` contains an unregistered category prefix
- `scenario.board` is present but has incorrect card count for the street
- `steps` is present but has fewer than 2 entries
- `difficulty` is outside 1-5 range
- `answer_by_pool` has a key other than "A", "B", or "C"

### Validation timing

- **Content load time** (content-loader): Full Zod validation against the canonical schema
- **Authoring time** (future): CLI lint command to validate drill JSON files before commit
- **Runtime**: No re-validation — trust the content pipeline

## 15. Downstream System Dependencies

### CLI Drill Engine

**Reads**: `drill_id`, `prompt`, `options`, `answer` (resolved by pool), `answer.required_tags`

**Adaptation needed**: The CLI currently uses option key letters ("A", "B", "C"). It needs to be updated to display `options[].label` and match user input against `options[].key` (action names). Alternatively, a thin adapter maps canonical option keys to positional letters for display.

### Table Sim

**Reads**: `scenario.board`, `scenario.hero_hand`, `scenario.hero_position`, `scenario.villain_position`, `scenario.pot_size_bb`, `decision_point`, `options`, `answer` (resolved by pool), `answer.correct_size`

**Adaptation needed**: Minimal. Field paths change (`meta.hero_pos` -> `scenario.hero_position`). The Table Sim needs an adapter or direct migration to read the canonical format.

### Scoring Engine

**Reads**: `answer.correct`, `answer.accepted`, `answer.required_tags`, `answer.correct_size`

**Adaptation needed**: The `scoreDrill()` function signature already accepts a `DrillAnswer` interface. The canonical `Answer` type is compatible with minor field renames. Pool resolution happens before scoring — the scoring engine always receives a resolved `Answer`, never an `AnswerByPool`.

### Adaptive Curriculum

**Reads**: `tags[]` (classification tags), `answer.required_tags` (rule tags), `difficulty`

**How it uses them**: Groups attempts by classification tag to detect weaknesses (e.g., "accuracy on `concept:blocker_effect` drills is below 50%"). Uses `difficulty` to calibrate study intensity.

### AI Coach

**Reads**: Full drill object — `prompt`, `scenario`, `answer` (resolved + all pool variants for contrast), `tags`, `coaching_context`

**How it uses them**: Constructs a coaching prompt with the player's wrong answer, the correct answer with explanation, the scenario context, and population-specific rationale. When `answer_by_pool` is present, contrasts answers across pools.

### Analytics

**Reads**: `tags[]`, `answer.required_tags`, `difficulty`, `node_id`, `scenario.street`, `scenario.pot_type`

**How it uses them**: Aggregates attempt accuracy by classification tag, street, pot type, and node. Identifies leak patterns (e.g., "low accuracy on `street:river` + `decision:bluff_catch`").

### Future Session Logger

**Writes**: Creates drills in canonical format from logged hands. Sets `metadata.source: "session_import"` and `metadata.source_hand_id` to link back to the original hand. The scenario model captures all the information a hand logger would record.
